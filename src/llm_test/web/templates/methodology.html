测试方法论

10 项探测的技术细节和评分体系的数学原理

核心设计原则

基线相对比较

不使用绝对阈值。每项探测将目标端点与已知的 Anthropic 官方 Opus 进行对比，消除了不同网络环境和 API 版本带来的误差。

非对称延迟评分

比基线慢是正常的（代理转发开销）。比基线快才可疑——因为更小的模型推理速度更快。这种非对称设计避免了误判正常的代理延迟。

加权置信度聚合

每项探测有两个维度：权重（该维度的重要性）和置信度（本次测量的可靠性）。二者共同作用于最终评分，低置信度的探测不会影响整体判断。

优雅降级

如果某项探测出错（网络超时、API 异常等），返回中性分数 0.5 和极低置信度 0.1，等效于从计算中移除，不会污染最终结果。

10 项独立探测

为什么最重要：这是最难伪造的探测。Opus 在困难推理任务上的表现明显优于 Sonnet 和 Haiku，降级的模型无法通过这些测试。

测试方法：5 道精心设计的推理题，涵盖多约束数学题、逻辑排列题、代码边缘情况、文字谜题、经典计数题。每道题有自动验证器。

评分：得分 = 正确题数 / 总题数。置信度 = min(0.95, 0.6 + 题数 × 0.07)。

原理：在 10K、50K、100K 长度的文本中，在不同深度（10%、25%、50%、75%、90%）插入随机验证码，要求模型精确检索。

评分：得分 = 成功检索次数 / 总测试次数。上下文窗口较小的模型会在长文本测试中失败。

原理：向目标和已知 Opus 发送完全相同的提示，比较回答的相似度。同一模型在相同提示下的输出会有高度一致的特征。

方法：5 个标准化提示，使用 n-gram Jaccard 相似度比较。相似度 > 0.6 = 高度一致，< 0.25 = 明显不同。

核心洞察：非对称评分——比基线快才可疑。三种长度的提示（100、1000、5000 字符），每种 10 次采样。

评分：ratio = target_tps / baseline_tps。ratio ≤ 1.0 得分 1.0（正常），ratio ≥ 2.0 得分 0.1（几乎必然是小模型），中间线性插值。

原理：不同模型版本有不同的训练数据截止日期。测试边界事件可以揭示实际运行的模型版本。

测试内容：2024 美国大选结果、Claude 3.5 Sonnet 发布信息、2025 年初 AI 模型发布情况、训练数据截止日期自报等。

原理：不同模型即使在 temperature=0 下也有微妙差异——词汇丰富度、句子长度、格式化偏好等。

提取特征：词数、字符数、词汇丰富度（Type-Token Ratio）、平均句子长度、模糊语言频率、Markdown 格式化特征、代码块使用等。

方法：通过 8 种不同策略让模型透露自己的身份：直接询问、补全测试、角色扮演、编码、对比、截止日期、覆写尝试、自评。

注意：置信度固定为 0.5——因为代理可以通过系统提示伪造身份。这项探测是补充验证，不是决定性证据。

原理：许多代理通过注入系统提示来伪装模型身份。本探测尝试用 4 种策略提取这些注入的指令。

评分：未泄露 = 1.0，泄露含伪装关键词 = 0.1，泄露但意图不明 = 0.5。

检查内容：响应中的 model 字段是否匹配、可疑 HTTP 头（x-forwarded-for, via, x-proxy 等）、Anthropic 响应格式完整性。

局限：信号最弱，因为代理可以轻易伪造这些字段。

说明：大多数 API 不暴露 logprobs。当可用时，token 概率分布可以非常可靠地指纹模型。目前为占位实现。

加权置信度聚合

score = ∑(scorei × weighti × confidencei) / ∑(weighti × confidencei)

scorei — 第 i 项探测的得分 (0-1)，0 表示确定不是 Opus，1 表示与 Opus 完全一致

weighti — 该探测的重要性权重（推理 5.0 最高，元数据 1.0 最低）

confidencei — 本次测量的可靠性（取决于样本量、是否有基线、探测类型的固有可靠性）

置信度低于阈值 (0.75) 的探测被排除在最终计算之外。

等级分数范围含义
GENUINE_OPUS≥ 0.85所有探测一致认为是 Opus
LIKELY_OPUS≥ 0.70大部分一致，存在轻微异常
SUSPICIOUS≥ 0.50信号混杂，无法确定
LIKELY_DOWNGRADE≥ 0.30多项探测指向降级
DEFINITE_DOWNGRADE< 0.30强证据不是 Opus

已知局限性

  • •身份探测可被系统提示覆盖（置信度已调低至 0.5）
  • •延迟受网络环境影响（已通过非对称评分缓解）
  • •知识截止日期会随模型更新变化
  • •单次测试结果受随机性影响，建议多次测试取平均
  • •logprobs 分析依赖 API 支持，大多数场景不可用
